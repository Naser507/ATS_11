The ATS_11 workbench was created to serve as a robust, automated, and cross-platform development environment specifically designed to streamline the building, packaging, and release of Windows GUI applications from a Linux host. The goal of the project was to construct a fully functional workflow that minimizes human error, automates repetitive tasks, and provides a reusable template for future projects, while also serving as a sandbox for testing new ideas safely.

The first step in creating this workbench involved designing a clean and modular project structure. The root directory was organized into folders such as client for source code, assets for images, sounds, and other resources, scripts for automation scripts, config for optional configuration files, and a dedicated z_Extras folder to store documentation, guides, and templates. Each folder was carefully planned to separate responsibilities, making it easy to locate code, resources, or scripts. Additional folders like build were used to capture outputs of debug and release builds, and Release_beta for final packaged releases.

One of the core challenges was cross-compiling Windows executables from Linux. To address this, we used MinGW-w64 13-posix as the compiler, and wxWidgets 3.2.6 as the GUI framework, compiled statically. The build process was orchestrated through a build.sh script, which handled compilation, linking, asset deployment, and DLL dependency collection. Special care was taken to ensure all required system and library dependencies were correctly identified using objdump and copied into the release folder, making the final ZIP fully portable. The script also allowed optional customization of the project name, enabling multiple variants or testing builds without renaming source files manually.

The packaging process was automated through package_release.sh, which copied the compiled binaries and assets to a temporary release folder. We incorporated a feature to add release notes, allowing users to append notes directly in the terminal, or use editors like nano, vim, or the system’s default editor. Notes were automatically saved into assets/release_notes with timestamps, ensuring proper documentation for each release. This step not only facilitated better project tracking but also created a record of changes and test results associated with each build.

To further streamline the workflow, a full_delivery.sh script was created to run the three main scripts — build, package, and upload — sequentially. This made full end-to-end testing and delivery simple, while keeping individual scripts modular for maintenance and debugging. The full delivery system also integrated with GitHub releases, automatically incrementing version tags and uploading assets, further reducing manual intervention.

Throughout the development of the workbench, careful attention was given to maintainability and clarity. Using mkdir -p ensured that folders were created safely without errors, even if the temporary release folder had been removed. The scripts were designed to be idempotent, meaning repeated runs would not break the workflow. Each script could run independently for testing, debugging, or partial workflow execution, which greatly enhanced reliability during development.

Lessons learned included the importance of folder creation order, robust error handling, and interactive prompts that enhance usability without compromising automation. By separating concerns across multiple scripts, we avoided creating a monolithic system that would be harder to maintain. We also ensured cross-platform awareness, making the build and release process reproducible on Linux while producing fully functional Windows executables.

Finally, the workbench now serves as a reusable template for future projects. Future improvements could include automating asset dependency tracking, improving editor fallback options for headless environments, adding detailed logging, and including versioning metadata in release notes. The system ensures that adding new source files, libraries, or assets is straightforward, reducing the risk of errors and significantly speeding up development cycles.

In conclusion, the ATS_11 workbench is a fully automated, cross-platform, and modular build environment that allows developers to compile, package, and release GUI applications with minimal effort. It captures best practices in automation, organization, and documentation, providing a reliable framework for future development projects and serving as a valuable learning tool for system design, build automation, and release management.

\section{Introduction}

This document provides a comprehensive guide to creating the \textbf{ATS\_11 workbench} — a fully automated, cross-platform, GUI-focused build and release environment for Windows applications developed from a Linux environment. The workbench is designed to be robust, modular, and reusable for future projects. 

It captures all design decisions, challenges, and solutions implemented in the workflow.

\section{Project Setup}

\subsection{Directory Structure}

\begin{lstlisting}
ATS_11/
├── client/              # Source code for GUI and application
├── build/               # Output of builds (binaries)
├── scripts/             # Build, package, and release automation scripts
├── assets/              # Images, sounds, and other resources
├── config/              # Optional configuration files
└── z_Extras/            # Extra documentation, templates, guides
\end{lstlisting}

Key points:
\begin{itemize}
    \item \texttt{client/windows/src/} contains main C++ GUI source code.
    \item \texttt{assets/} stores images, logos, and sounds.
    \item \texttt{z\_Extras/} is reserved for meta-information, templates, and workflow guides.
\end{itemize}

\subsection{Version Control}

\begin{itemize}
    \item Git is used to track all project files.
    \item GitHub repository uses automated release handling with version tags (e.g., \texttt{v0.x.x}).
    \item \textbf{Recommendation:} Enable 2-factor authentication for secure release uploads.
\end{itemize}

\section{Building the Environment}

\subsection{Dependencies}

\begin{itemize}
    \item \texttt{wxWidgets 3.2.6} compiled statically for MinGW 64-bit.
    \item MinGW-w64 13-posix compiler on Linux.
    \item Linux commands: \texttt{mkdir}, \texttt{rm}, \texttt{cp}, \texttt{objdump}, \texttt{zip}.
    \item Editors: \texttt{nano}, \texttt{vim}, or system default for notes.
\end{itemize}

\subsection{Build Script (\texttt{build.sh})}

The build script:
\begin{itemize}
    \item Compiles the GUI and main application executable.
    \item Prompts for a custom or default project name.
    \item Sets up output directories: \texttt{build/client/release/}, \texttt{Release\_beta/[project\_name]/bin/}, \texttt{Release\_beta/[project\_name]/assets/}.
    \item Compiles using MinGW with static wxWidgets linking.
    \item Resolves DLL dependencies automatically using \texttt{objdump}.
    \item Copies all assets and configuration files to the release folder.
    \item Packages everything into a release ZIP.
\end{itemize}

\subsection{Key Features}

\begin{itemize}
    \item Modular design: new source files or libraries can be added easily.
    \item Cross-compilation: builds Windows executables from Linux.
    \item Full automation of DLL handling and asset packaging.
\end{itemize}

\section{Packaging and Release Automation}

\subsection{Package Script (\texttt{package\_release.sh})}

\begin{itemize}
    \item Copies executables and assets to a clean temporary release folder.
    \item Resolves required DLLs automatically.
    \item Prompts for release notes:
        \begin{itemize}
            \item Append notes directly in terminal.
            \item Edit using \texttt{nano}, \texttt{vim}, or system default editor.
            \item Notes are saved in \texttt{assets/release\_notes/} within the release folder.
        \end{itemize}
    \item Creates release ZIP including binaries, assets, and notes.
    \item Designed to integrate with \texttt{full\_delivery.sh}.
\end{itemize}

\subsection{Release Notes Management}

\begin{itemize}
    \item Notes are timestamped: \texttt{release\_notes\_YYYY-MM-DD\_HH-MM.txt}.
    \item Ensures folder existence using \texttt{mkdir -p} to prevent permission errors.
\end{itemize}

\section{Full Delivery Script (\texttt{full\_delivery.sh})}

\begin{enumerate}
    \item Run \texttt{build.sh} — compile executable and setup release folder.
    \item Run \texttt{package\_release.sh} — copy files, handle notes, zip release.
    \item Run \texttt{upload\_release\_assets.sh} — create/update GitHub release and upload assets.
\end{enumerate}

\subsection{Design Considerations}

\begin{itemize}
    \item Minimalist orchestration improves maintainability.
    \item Each script can run independently for testing or debugging.
\end{itemize}

\section{Lessons Learned}

\begin{itemize}
    \item Folder creation order matters: TEMP\_RELEASE must exist before creating notes folder.
    \item Use of \texttt{mkdir -p} ensures idempotent folder creation.
    \item Separation of concerns improves maintainability and clarity.
    \item Interactive prompts enhance usability without breaking automation.
    \item Cross-platform awareness is critical: Windows executables built from Linux.
\end{itemize}

\section{Recommended Practices}

\begin{itemize}
    \item Verify library and DLL paths before packaging.
    \item Test full delivery workflow after adding new source files or assets.
    \item Maintain \texttt{z\_Extras} for documentation and workflow templates.
    \item Keep scripts modular rather than monolithic.
\end{itemize}

\section{Future Improvements}

\begin{itemize}
    \item Automate dependency scanning for new asset types (fonts, sounds).
    \item Improve default editor fallback for headless environments.
    \item Add detailed logging for build/package/upload steps.
    \item Include automatic versioning metadata in release notes.
\end{itemize}

\section{Conclusion}

The ATS\_11 workbench is now a robust, reusable template for building, packaging, and releasing GUI applications. Following this guide ensures the same workflow can be replicated or extended for future projects.
